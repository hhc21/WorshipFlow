# WorshipFlow Product Roadmap (2026-03-10 Baseline)

## 0. 문서 역할

`product_roadmap.md`는 제품 전략 문서다.

- 구현 계획/체크리스트: `plan.md`
- 코드 분석/현황: `research.md`
- 제품 방향/우선순위/백로그: `product_roadmap.md`

즉, 이 문서는 "무엇을 왜 만들지"를 정의한다.

## 1. 현재 제품 방향

### 1.1 현재 개발 운영 모드

- 현재 사이클은 **웹 기준 개발을 지속**한다.
- 모바일 실기 검증은 즉시 확장하지 않고, 배포 게이트 시점으로 이연한다.

보류 항목(배포 직전/배포 후 내부 테스트에서 재개):
- iOS 로컬 Google 로그인 이슈 재검증
- iPad Apple Pencil 실기 필기 검증
- iOS 공유 필기 실기 검증

### 1.2 제품 핵심 축

WorshipFlow의 핵심 시스템은 여전히 `LiveCue`다.

- 팀 운영에서 실제 가치가 발생하는 지점은
  - 악보 로드 정합성
  - 실시간 동기화
  - 필기 입력 안정성
  - 운영자/팀 동선 단순화
- 이번 기준선에서 운영자 → 팀 진입 경로가 복구되며, 운영 도구 사용성이 제품 축에 포함된다.

## 2. 제품 구조 관점 우선순위

### 2.1 Now (현재 구현/운영 범위)

- LiveCue 핵심 흐름 안정화 유지
- 운영자 도구에서 팀 워크스페이스로 진입 가능
- 팀 기반 프로젝트 운영(팀 선택 → 팀 홈 → 프로젝트)
- 웹 fallback/보조 뷰어(Next.js Viewer) 유지

### 2.2 Next (다음 사이클 우선)

- 운영자 경로의 팀 진입 이후 동선(읽기/관리)의 일관성 보강
- LiveCue 사용자 체감 품질 개선(도구 전환, 렌더 체감, 장시간 세션 안정성)
- 데이터/권한 문서와 실제 런타임 경로 정합성 강화

### 2.3 Later (중장기 백로그)

- 모바일 실기 증빙 재개(배포 게이트)
- 플랫폼별 운영 품질 지표 자동 수집 강화
- 멀티 조직(예: churches 축) 확장 가능성 검증
- 운영/분석용 백오피스 기능 체계화

## 3. 운영자 모드(Product Position)

운영자 모드는 기능 데모가 아니라 실제 운영 안정화 도구다.

현재 위치:
- 글로벌 운영자가 팀 목록을 확인
- 특정 팀 워크스페이스로 진입
- 팀 단위 운영 상태를 확인

향후 확장 원칙:
- 운영자 모드는 "팀 워크스페이스 접근/복구/관리" 중심으로 확장한다.
- 팀 사용자 경로를 우회해 별도 기능 체계를 만들기보다, 팀 경로와 정합한 관리 계층을 유지한다.

## 4. LiveCue Product Direction

### 4.1 고정 원칙

- LiveCue는 제품의 핵심 경험이다.
- "곡 로드/동기화/필기" 3축 안정성이 릴리즈 품질을 결정한다.

### 4.2 Next.js Viewer 위치

- Next.js Viewer는 핵심 엔진이 아니라 웹 fallback/보조 경로다.
- 유지 여부는 신념이 아니라 실측 데이터로 판단한다.

판단 기준:
1. Flutter 단일 경로의 재현 이슈 존재 여부
2. Viewer 경로의 재현율 완화 효과
3. 이중 유지 비용 대비 안정성 이득

## 5. 출시 품질 관점

### 5.1 현재 수준

- 구조 분리(Sync/Render/Input/Persistence) 반영
- SP-04 하드닝 반영
- 운영자 → 팀 진입 경로 반영

### 5.2 아직 필요한 제품 증빙

- 실기기 최종 증빙(iPad/iPhone)
- 보류된 iOS 인증/공유 필기 검증
- 운영자 경로 확장 시 역할/권한 경계의 제품 UX 명확화

## 6. 로드맵 운영 규칙

1. 제품 전략과 구현 계획을 혼합하지 않는다.  
2. 구현 완료와 제품 검증 완료를 구분한다.  
3. 보류 항목은 "미완료"가 아니라 "재개 시점이 정의된 대기"로 관리한다.  
4. 문서 기준선은 코드와 충돌하지 않게 유지한다.

---

현재 제품 로드맵의 핵심 요약:
- 단기: 웹 기준 개발 지속 + LiveCue 안정화 유지 + 운영자 팀 진입 경로 정착
- 중기: 배포 게이트 시점에 모바일 실기 검증 재개
- 장기: 데이터 기반으로 fallback 경로 비중을 재평가하고 플랫폼 전략을 확정
